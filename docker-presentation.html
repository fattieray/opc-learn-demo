🤔 传统部署的痛点

😰

环境不一致

"在我电脑上能跑啊！"
开发、测试、生产环境不同

🔧

配置复杂

安装 Node.js、Python、Nginx
配置环境变量、依赖冲突

💥

难以迁移

换服务器就要重新配置
耗时长，容易出错

⚡ 传统部署 vs Docker 部署

❌ 传统方式
  • ✗ 手动安装所有依赖
  • ✗ 环境配置容易出错
  • ✗ 不同服务器结果不同
  • ✗ 迁移困难
  • ✗ 占用资源多
✅ Docker 方式
  • ✓ 一键构建镜像
  • ✓ 环境完全一致
  • ✓ 随处运行
  • ✓ 秒级部署
  • ✓ 资源隔离

🎯 Docker 四大核心概念

📦

镜像 Image

应用的打包模板
包含代码、依赖、配置

📱

容器 Container

镜像的运行实例
轻量、隔离、可复用

💾

数据卷 Volume

持久化数据存储
容器删除数据不丢失

🌐

网络 Network

容器间通信桥梁
安全隔离又互联

💡 通俗理解 Docker

🍔 类比：快餐连锁店

镜像 = 标准化食谱和食材包

容器 = 每家分店的厨房

数据卷 = 冰箱（保存食材）

网络 = 店内通讯系统

🎮 类比：游戏机

镜像 = 游戏卡带

容器 = 运行中的游戏

数据卷 = 存档文件

网络 = 多人联机功能

核心思想：一次打包，随处运行！

📝 Dockerfile 示例

FROM node:18-alpine
WORKDIR /app
COPY . .
RUN npm install
CMD ["npm", "start"]

docker build -t myapp .
docker run -p 3000:3000 myapp

🎼 Docker Compose 编排

一个文件管理多个容器：

version: '3.8'
services:
  backend:
    build: ./backend
    ports: ["9000:9000"]

  frontend:
    build: .
    ports: ["3000:3000"]

  nginx:
    image: nginx:alpine
    ports: ["80:80"]

一条命令启动所有服务：

docker-compose up -d

🚀 实际部署步骤

1构建镜像

在本地电脑打包应用

docker build -t opc-frontend .
docker build -t opc-backend -f backend/Dockerfile .

2导出镜像

保存为文件方便传输

docker save opc-frontend opc-backend > images.tar

3上传服务器

传输到腾讯云

scp images.tar root@124.221.40.35:/tmp/

4启动服务

一键部署完成

docker load < images.tar
docker-compose up -d
